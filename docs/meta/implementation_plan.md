# Implementation Plan — Phase 5: Operational Playbooks, Runbooks & SLA Enforcement

This implementation plan details the programmatic enforcement of the operational playbooks, runbooks, and SLA boundaries defined in the Phase 5 specification. It transforms the CIC/MAAL system into an operationally secure control plane with automated safety gates, persistence, and cryptographic lineage integrity.

---

## User Review Required

> [!IMPORTANT]
> **State Persistence Location:**
> We propose maintaining the state of SLA metrics, active playbooks, and lockdown freezes in a JSON file `governance/cicState.json` rather than in-memory. This enables the CLI, the API server, the ingestion loop, and the governance orchestrator to share the same state and persist it across service restarts.

> [!WARNING]
> **Pre-Existing Test Failures resolved:**
> 1. We identified a syntax error during `uuid` ESM import inside `node-cron` because Jest was resolving the browser bundle in JSDOM environment. We propose mapping `^uuid$` to the Node-compatible CommonJS build in `jest.config.js`.
> 2. We identified a cache policy test failure where `data-adapter` is invoked under `READ_ONLY` policy on a clean cache, causing a cache miss. We propose populating the cache first before switching the policy to `READ_ONLY` to correctly test that `READ_ONLY` serves from cache.
> 3. We identified that `invalidateAdapter` does not remove cache entries when no patterns are configured. We will track keys generated by each wrapper in a set and invalidate all tracked keys on default invalidation.

---

## Open Questions

- **Q1: Frequency of SLA violation re-evaluations**
  Should SLA violation detectors run strictly on the 30s ingestion loop cycle, or on every incoming request to `/v1/chat`? We propose evaluating request-level SLAs (latency, token usage) on every turn and loop-level SLAs (backlog, routing stability, governance hash integrity) on the 30s ingestion loop.

### Component 4: Governance Hooks & Cryptographic Hash-Chain

#### [MODIFY] [audit-policy.ts](file:///c:/dev/governance/audit-policy.ts)
- Modify `appendAuditEvent()` to read the previous event's hash and incorporate it into the current event's `hash` calculation (proper cryptographic hash-chain).
- Add new event types: `"drift_decay"`, `"drift_score_updated"`, `"sla_violation"`, `"governance_lockdown"`, `"promotion_frozen"`.
- Implement `verifyAuditChain(logPath)`: traverses the events list and validates the continuity of the hashes.
- If a chain break is detected, log a critical SLA violation and write to the state to trigger the Governance Lockdown Runbook.

#### [MODIFY] [promotion-rollback.ts](file:///c:/dev/governance/promotion-rollback.ts)
- In `executePromotion` and `executeRollback`, load `governance/cicState.json`.
- If `promotionsFrozen` or `rollbacksFrozen` is active, reject the request with `ERR_PROMOTIONS_FROZEN` or `ERR_ROLLBACKS_FROZEN`.

## Verification Plan

### Automated Tests

- Create new test suite `src/tests/phase5-operational.test.ts` containing 40+ tests verifying:
  - **SLA violations**: Latency, tokens, backlog, routing oscillation, and hash-chain break detection.
  - **Playbook actions**: Drift Spike Playbook (decay + pruning), Routing Stability Playbook (freezing routing), Backend Recovery Playbook.
  - **Runbooks**: Lockdown runbook gates, Ingestion recovery JSONL parsing, idempotency, and backlog thresholds.
  - **Drift decay**: Verifying 5% decay per cycle is logged in audit trail.
  - **Ingestion Daemon**: Verify line-by-line streaming, error recovery on corrupted JSON, and idempotency.
- Run all tests:
  ```bash
  npm test
  ```

### Manual Verification

- Run the adapter gateway:
  ```bash
  npm run start:adapter-gateway
  ```
- Corrupt a line in `client_sessions.jsonl` to test that the Ingestion Daemon skips it and continues.
- Tamper with a hash in `governance/audit-log.json` to verify that the next ingestion run detects the hash-chain break, triggers SEV-1 Governance Lockdown, and freezes promotions.
