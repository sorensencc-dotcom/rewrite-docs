==============================================================================
Phase 33: Federated KG (three-backend architecture)
==============================================================================

  [TorqueQuery]  (temporal events)
     (Phase 26)
        |
        v
  [FederatedKGQueryEngine]
        |
        +--> [Postgres KG]        [SQLite KG]        [Qdrant]
             (structure)          (audit/replay)     (semantic)
             - nodes              - digest chain     - embeddings
             - edges              - determinism      - vectors
             - multi-writer       - replay safety    - similarity
             - high-throughput                       - ranking

Storage roles (Phase 33):
  - SQLite: audit + replay + digest chain (append-only, deterministic)
  - Postgres: authoritative structural KG (nodes, edges, temporal)
  - Qdrant: semantic overlay (embeddings, similarity search)
  - TorqueQuery: event index (temporal, causal ordering)

Query pattern:
  1. Structural filter → Postgres
  2. Semantic ranking → Qdrant (if applicable)
  3. Temporal ordering → TorqueQuery
  4. Merge + rank results

Example: Find nodes semantically similar to "API performance" within last 30 days
  - Postgres: SELECT nodes WHERE type='metric' AND valid_from > now()-30days
  - Qdrant: SELECT node_ids ORDER BY embedding_distance(embedding, "API performance")
  - Merge: nodes that satisfy both criteria, ranked by Qdrant distance

==============================================================================
Key Design Decisions
==============================================================================

1. SQLite remains append-only after Phase 31
   → Digest chain integrity is immutable
   → Replay + audit trail preserved for history

2. Dual-write avoids split-brain during migration
   → Both stores see identical mutations
   → Safe to switch reads at any time

3. Qdrant is semantic sidecar, not primary store
   → KG structure remains in Postgres
   → Embeddings are derived data, not authoritative
   → Can rebuild Qdrant from Postgres if needed

4. Phase 33 federation is read-optimized
   → Structural queries → Postgres
   → Semantic queries → Qdrant
   → Temporal queries → TorqueQuery
   → No write path changes in Phase 33

5. Zero downtime for each migration
   → New backend deployed alongside old
   → Reads switched via config flag
   → Writes dual-enabled until old store deprecated
